# Innerlife

Innerlife 是一个实验性的本地虚拟人格运行时，用来探索虚拟人如何拥有长期情景记忆、实体图召回、情绪状态、关系状态、后台整合，以及可观测的内部运行过程。

它不是生产级多用户聊天平台，而是面向长期记忆、虚拟人格和智能体运行时研究的开发原型。（目前仍处于实验阶段）

## 项目亮点

Innerlife 的核心亮点不是单个聊天界面，而是一套围绕虚拟人格持续运行而设计的模块系统。它把一个虚拟人拆成可组合、可观察、可持续更新的多个部分，让人格、记忆、情绪、关系、工具和后台进程各自承担明确职责。

在这个设计里，虚拟人格不只是名字和人设，也不只是提示词加聊天历史，而是由多个模块共同构成的运行实体：

- 人格模块
- 记忆模块
- 情绪模块
- 关系模块
- 工具模块
- 后台进程
- 观测器

另一个重点是实体图记忆设计。系统不只按文本相似度检索记忆，而是把人、地点、物品和事件组织成实体节点，并允许每个实体拥有个人化别名。这样可以模拟人类对具体对象的稳定心理表征，也能让一次记忆命中沿着权重边扩散到相邻实体，带出一片相关经历。

这套结构让虚拟人格可以逐步形成自己的记忆网络、关系状态和内部节律，而不是每轮对话都从零开始临时扮演。

## 当前能力概览

- 多虚拟人格创建、编辑、删除
- 网页聊天界面
- 流式回复和中断
- Anthropic、OpenRouter、OpenAI 兼容模型服务商
- 本地数据库持久化
- 虚拟人格级工具管理
- 短期记忆检索
- 实体图 + 长期情景记忆召回
- 情绪状态系统
- 关系状态系统
- 后台进程记忆整合
- 观测器调试界面
- 中英文界面切换

## 管理系统

Innerlife 不只是一个聊天窗口，也提供了完整的网页端管理系统。你可以在界面里创建和配置虚拟人格，调整模型服务商、人格提示词、工具权限、记忆系统、情绪状态和关系对象。

记忆管理页面可以查看和编辑短期记忆、情景记忆、实体节点、别名和无类型权重边；提示词测试面板可以直接运行每个记忆阶段的提示词，查看输入和输出是否符合预期。

这让 Innerlife 更像一个可以调试、塑形和维护的虚拟人格工作台，而不是只能聊天的演示应用。

## 模块说明

### 1. 人格模块

人格模块负责决定虚拟人的基本身份、说话方式和互动边界。

当前包括：

- 系统提示词
- 人格提示词
- 虚拟人格名称
- 虚拟人格描述
- 头像和资料信息

人格不是直接写死在主提示词里，而是作为虚拟人格配置被加载，并在每轮对话时进入运行时。

### 2. 记忆模块

记忆模块是当前项目最核心的部分。

Innerlife 不把长期记忆简单等同于聊天记录，也不把所有记忆都直接塞进提示词。它把记忆拆成几层，并让不同层承担不同职责：

```
活跃上下文
  -> 短期记忆
  -> 情景记忆
  -> 实体图
```

#### 活跃上下文

活跃上下文是当前活跃对话窗口，代表“最近正在聊的内容”。它会参与当前对话，但不会直接变成长期记忆。

#### 短期记忆

短期记忆保存近期有价值的信息。每轮聊天前，系统会先检索短期记忆，再决定要不要把命中的记忆放进提示词。

短期记忆检索不是简单关键词搜索，而是由两个分析器共同生成本轮检索意图：

- **时间分析器**：每轮聊天前调用语言模型，根据当前时间、最近对话和当前用户消息，解析适合记忆检索的时间范围；失败时回退到本地时间解析器。
- **语义分析器**：一次轻量语言模型调用，用最近几条对话补全当前问题里的代词、省略和回指，生成一句稳定的检索查询。

随后系统把时间范围和语义查询合并，再用向量检索已有短期记忆：

```
当前用户消息
  -> 时间分析器
  -> 语义分析器
  -> 向量检索
  -> 命中的短期记忆进入提示词
```

如果短期记忆已经足够回答，主模型可以直接使用短期记忆，不需要再调用长期记忆工具。

#### 情景记忆

情景记忆保存长期事件、场景、关系和经历。它不是逐轮聊天摘要，而是后台进程从短期记忆中整理出的更稳定记忆片段。

一条情景记忆通常包含：

- 摘要
- 详细内容
- 重要性
- 观测时间范围
- 绑定实体
- 摘要向量

其中摘要用于向量化和检索，详细内容是真正返回给主模型阅读的记忆内容。

写入长期情景记忆时，后台进程会先从短期记忆中抽取局部实体和情景记忆草稿，再把局部实体解析到全局实体图：

```
短期记忆
  -> 局部实体 + 情景记忆草稿
  -> 实体解析
  -> 情景记忆 + 实体绑定
```

这样做的目的是避免过早把原文里的称呼标准化，同时又能让长期记忆最终挂到稳定实体节点上。

#### 实体图

实体图的设计来自一个很简单的观察：人类记忆并不是只按文本相似度存放的。我们会为具体的人、地点、物品和事件形成稳定的心理表征，并且每个人都会给这些对象建立自己的称呼、昵称和别名。比如同一个游戏、同一个人、同一只猫，在不同人的记忆里可能有完全不同的叫法。

另一方面，人类回忆也不是单点命中。想到一个人，往往会顺带想起相关地点、共同经历、常聊的话题和附近的情绪背景。这类似联想记忆里的激活扩散：一个节点被点亮后，会沿着相关连接扩散到附近节点，再带出一片相关记忆。

所以实体图在这里不是为了做知识图谱，而是为了模拟“个体化称呼”和“联想式回忆”：别名解决同一个对象的不同叫法，权重边解决记忆从一个实体扩散到相邻实体的问题。

实体图保存记忆中的稳定对象和它们之间的连接。实体类型目前收窄为：

- 人物
- 地点
- 物品
- 事件

每个实体可以有标准名称、描述和别名。别名不会在普通抽取阶段随便生成，只会在实体解析判断“这是同一个对象”时建立。

实体之间的边是无类型权重边，表示它们在情景记忆中共同出现的强度，而不是手写的关系类型。

长期召回时，实体图负责把一次命中扩展成一片相关记忆网络：

```
实体提及
  -> 标准名称 / 别名匹配实体节点
  -> 点亮命中实体
  -> 一跳扩散到相邻实体
  -> 找到相关情景记忆
```

这使得系统既可以通过“星河2”这样的别名找到“星河战术2”节点，也可以通过被点亮节点附近的实体召回相关情景记忆。

#### 长期记忆召回

长期记忆不会每轮自动进入提示词，而是通过长期记忆搜索工具召回。召回时会同时走实体图和文本语义两条路：

```
当前问题 + 最近上下文
  -> 抽取实体提及
  -> 标准名称 / 别名匹配实体节点
  -> 一跳权重扩散
  -> 摘要向量语义匹配
  -> 返回相关情景记忆详细内容
```

工具返回的是情景记忆的详细内容，而不是只返回摘要。被召回的情景记忆还会获得一个短期激活状态，在接下来一段时间里参与短期记忆检索；它不会被永久固化进提示词。

### 3. 情绪模块

情绪模块维护虚拟人格的持续情绪状态。

当前维度：

- 心情
- 精力
- 压力

每轮对话后，系统会分析这轮互动对情绪的影响，并写入新的情绪状态。

情绪会随时间或轮次向基线状态衰减，不是每轮完全重新生成。

情绪状态会以提示词片段的形式影响回复语气，但不会直接暴露分数给用户。

### 4. 关系模块

关系模块维护虚拟人格面向不同对象的关系状态。它的目标不是让模型每轮临时“扮演关系”，而是让关系作为持续状态被读取、注入、分析和更新。

当前维度：

- 信任
- 亲近
- 熟悉
- 尊重

目前有两种模式。多维关系是默认单对象关系，适合只关心“虚拟人格与用户”这一条关系的简单场景。

更有特色的是命名多维关系：同一个虚拟人格可以维护多个命名关系对象，每个对象都有独立的关系状态和历史。

一个关系对象可以包含：

- 名称
- 角色
- 描述
- 头像
- 主观备注
- 关系维度
- 关系历史

聊天时，当前会话可以绑定到某个关系对象。只有绑定后，关系模块才会把这个对象的关系状态注入提示词：

```
会话
  -> 绑定关系对象
  -> 读取关系状态
  -> 注入关系提示片段
  -> 分析本轮关系变化
  -> 只更新该关系对象
```

这意味着同一个虚拟人格面对不同对象时，可以拥有不同的亲疏、信任、熟悉度和尊重感。比如它可以同时维护“研究搭档”“普通访客”“长期朋友”等不同关系，而不是把所有互动混成一个全局用户状态。

每轮对话结束后，系统会分析这轮互动对当前绑定对象的影响，生成关系增量，并把状态向基线状态做轻微衰减。关系状态会影响语气、耐心、亲近感和措辞，但不会直接把分数暴露给用户。

### 5. 工具模块

工具模块控制虚拟人格能主动使用哪些能力。当前说明重点关注长期记忆搜索工具。

长期记忆搜索工具是主模型访问长期情景记忆的入口。Innerlife 不会把所有长期记忆每轮都塞进提示词，而是让模型在上下文和短期记忆不够时主动调用这个工具。

它的特色是：工具调用本身不是简单文本搜索，而是会结合本轮短期记忆检索得到的语义查询、工具输入查询、最近上下文和实体图。

```
长期记忆搜索工具
  -> 生成长期记忆文本查询
  -> 抽取实体提及
  -> 标准名称 / 别名匹配实体节点
  -> 实体图一跳扩散
  -> 摘要向量语义召回
  -> 融合排序
  -> 返回情景记忆详细内容
```

如果召回成功，工具返回的是情景记忆的详细内容，让主模型能读到更完整的事件描述，而不是只看到短摘要。

被召回的情景记忆还会被短期激活一段时间。后续对话里，它可以像短期记忆一样被前置检索命中；但这种激活有数量上限和过期时间，不会变成永久固定提示词。

### 6. 后台进程模块

后台进程是 Innerlife 的本地后台心跳。

主聊天只处理“此刻的对话”，而后台进程负责在对话之外继续缓慢工作：检查活跃上下文、整理短期记忆、沉淀长期情景记忆，并记录后台事件。它让虚拟人格不只是被动等待下一句话，而是在本地进程中拥有一个持续整理自身状态的节律。

当前负责：

```
上下文 -> 短期记忆
短期记忆 -> 情景记忆 + 实体图
```

后台进程会把对话上下文逐步整理成结构化记忆，并写入本地数据库。它不会替主模型回答用户，也不直接改变正在进行的一轮回复；它处理的是那些适合在后台慢慢完成的记忆整合工作。

当前后台进程也会把自己的运行状态和事件流落库，例如启动时间、心跳、最近错误、刷新结果和睡眠整合结果。这样观测器和后台进程工作台可以看到这个“后台心跳”是否仍在运行。

### 7. 观测器模块

观测器是研究和调试入口，用来观察虚拟人格内部运行过程。

它既可以在独立观测器页面事后回放，也可以在聊天页面对话时实时打开观测抽屉，查看当前这一轮正在发生的语言模型调用、提示词片段、工具调用和记忆召回。

可以查看：

- 最终系统提示词
- 提示词片段
- 语言模型调用
- 工具调用
- 记忆检索细节
- 实体提及
- 被点亮实体
- 情景记忆命中
- 情绪分析
- 关系分析

它的目标是让“虚拟人为什么这样回答”变得可检查。

## 系统架构

```
apps/web
  Next.js 网页界面、接口路由、聊天页面、虚拟人格管理页面

packages/core
  语言模型服务商、智能体循环、工具执行、流式运行时

packages/systems
  系统生命周期模块：
  记忆、情绪、关系、上下文压缩

packages/db
  本地数据库表结构、迁移、数据仓库、初始化

packages/daemon
  本地后台任务：
  上下文 -> 短期记忆
  短期记忆 -> 情景记忆 + 实体图

packages/observer
  语言模型调用记录、提示词快照、内部过程观测
```

## 运行流程

```
用户输入
  -> 网页聊天接口
  -> 加载虚拟人格 / 会话 / 模块
  -> 轮次开始前的系统处理
  -> 调用模型前拼接提示词片段
  -> 核心语言模型和工具循环
  -> 模型回复后的系统处理
  -> 轮次结束后的系统处理
  -> 持久化消息、状态和后台任务数据
```

```
npm install
cp .env.example .env
npm run dev --workspace @mas/web
```

打开：

```
http://localhost:3000
```

可选启动后台进程：

```
npm run daemon:start
```

## 示例使用方案

`examples/` 目录下提供了两个干净的样例虚拟人格：

- `examples/zh` 是中文样例。
- `examples/en` 是英文样例。

每个样例都使用和应用运行时一致的本地数据结构：

```
storage/app/data.db
storage/memory/memory.db
```

如果要试用样例虚拟人格，建议先使用空的本地数据目录，或者先把你自己的 `storage/` 目录移到别处，再把某个样例复制到项目根目录：

```
cp -r examples/zh/storage ./storage
```

如果要试用英文样例，把路径换成 `examples/en/storage`。

推荐的体验方式：

- 先用中文样例虚拟人格，询问它近期记忆、关系对象和长期经历。
- 聊天时打开观测器抽屉，查看提示词片段、记忆检索、被点亮实体、工具调用、情绪分析和关系分析。
- 启动后台进程，让它把上下文整理成短期记忆，再沉淀成情景记忆和实体图。
- 如果想测试纯净架构，可以从空数据库开始新建一个虚拟人格。

## 模型服务商配置

当前支持：

- Anthropic
- OpenRouter
- OpenAI 兼容接口

记忆向量化当前主要通过 OpenRouter 调用，默认模型是：

```
qwen/qwen3-embedding-8b
```

配置分成两部分：

- 在 `.env` 里填写模型服务商密钥和可选接口地址。
- 在虚拟人格设置页选择聊天模型服务商和模型。
- 在记忆设置里选择记忆向量化服务商和向量模型。

Anthropic、OpenRouter、OpenAI 兼容接口和 DeepSeek 兼容接口的配置示例见 `.env.example`。

## 本地数据和隐私

默认数据存储在：

```
storage/app/data.db
storage/memory/memory.db
```

这些数据库可能包含真实聊天、提示词、记忆和关系状态。

不要把真实数据库提交到公开仓库。

仓库中的 `examples/` 数据库是用于公开演示的样例数据库。以后如果修改样例虚拟人格，发布前仍然需要重新检查里面没有私人内容。

## 当前限制

- 这是实验性项目，不保证接口和数据库表结构稳定。
- 当前主要面向本地单用户研究，不是生产级多用户服务。
- 记忆召回仍然是研究级实现，需要继续测试和调参。
- 本地数据可能包含敏感内容，公开前必须清理。
- 模型服务商、向量化、后台进程行为仍在快速演化。
