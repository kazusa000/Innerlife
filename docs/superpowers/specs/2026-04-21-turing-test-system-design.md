# 图灵测试系统 — 设计文档

日期：2026-04-21
状态：设计完成，待实现

---

## 1. 目标

创建一个**外部优先、异步运行**的图灵测试系统，用来自动评测某个虚拟 agent 的拟人感。

这个系统的核心不是页面，而是一个可被 Codex 或其他 AI 调用的测试任务系统。页面只是它的控制台。

一次图灵测试的目标结果是：

- 复制当前选中的 persona 配置
- 强制开启全部模块
- 创建一个临时测试 agent
- 由测试官按固定测试套件与该 agent 对话
- 必要时插入测试记忆或测试状态
- 如果出现极度不符合人的地方，立即中断整场评测
- 产出：
  - 主输出：评测报告
  - 副输出：完整对话回放与证据

这套系统测的不是“模型聪不聪明”，而是：

- 像不像一个持续存在的人
- 会不会记事
- 情绪与关系变化是否可信
- 会不会在关键时刻露出“AI 助手”破绽

---

## 2. 已确认的约束

### 2.1 外部优先

系统主体必须是一组可被外部 AI 调用的接口与异步任务能力，不先以页面逻辑为核心。

页面只做：

- 发起测试
- 查看进度
- 阅读报告
- 回看回放
- 清理测试数据

### 2.2 异步执行

图灵测试第一版必须是异步任务：

- 创建 run
- 后台执行
- 前端或外部 AI 轮询 / 订阅状态
- 完成后读取结果

不做同步阻塞式接口。

### 2.3 复制当前 persona，生成临时测试 agent

每次测试都基于当前选中的 persona：

1. 复制配置
2. 强制开启全部模块
3. 生成临时测试 agent

测试期间产生的聊天、Observer、memory、emotion、relationship 等数据全部挂在临时测试 agent 上，与正式 persona 隔离。

### 2.4 测试数据默认保留，但支持一键全删

测试跑完后默认保留：

- 临时测试 agent
- 测试对话
- Observer 数据
- memory / emotion / relationship 变化
- 评测报告
- 事件日志

同时系统必须支持“一键清理本次测试全部数据”。

### 2.5 固定测试套件

第一版不做自由编排测试流程。

固定使用 6 段测试套件，确保：

- 每次评测可比较
- 报告结构稳定
- 后续更容易做回归评测

### 2.6 固定内置规则书

测试官的强约束规则先内置在系统目录下的 markdown 文件中，不做运行时切换版本。

## 3. 系统总体结构

系统拆成 5 个核心对象：

### 3.1 Turing Test Run

一次完整的图灵测试任务。

它负责：

- 记录来源 persona
- 挂接临时测试 agent
- 跟踪执行状态
- 汇总报告、回放、日志、证据

### 3.2 Temporary Test Agent

从当前 persona 复制出来的测试专用 agent。

特点：

- 强制开启全部模块
- 与正式 persona 隔离
- 仅用于本次测试
- 默认保留，可清理

### 3.3 Fixed Test Suite

固定 6 段测试场景：

1. 自然开场
2. 日常延续
3. 记忆追问
4. 情绪合理性
5. 关系边界
6. 不确定性与露馅处理

### 3.4 Turing Judge Runner

真正执行评测的后台测试官执行器。

职责：

- 按固定套件推动对话
- 在固定节点插入测试记忆 / 测试状态
- 监听被测 agent 回复与 observer
- 判断是否触发红线
- 中断或完成测试
- 生成最终报告

### 3.5 Report + Transcript

一次 run 的最终结果分两层：

- **Report**
  - 拟人感评分
  - 关键失败点
  - 红线触发原因
  - 修复建议
- **Transcript**
  - 完整对话回放
  - 插入过的测试记忆 / 状态
  - 对应 observer 证据入口

## 4. 任务状态机

第一版状态机固定为：

- `queued`
- `preparing`
- `running`
- `interrupting`
- `interrupted`
- `completed`
- `failed`
- `cleaned`

含义：

- `queued`：run 已创建，等待执行
- `preparing`：复制 persona、创建临时测试 agent、强制开模块、装载规则书
- `running`：正在执行固定测试套件
- `interrupting`：已触发红线，正在停止并生成失败报告
- `interrupted`：被红线终止，报告已落地
- `completed`：正常完成全部测试段
- `failed`：系统错误或运行异常，不是评测失败
- `cleaned`：测试数据已被一键清理

---

## 5. 外部接口

第一版至少提供这 5 类外部接口。

### 5.1 创建测试任务

输入：

- `sourceAgentId`

动作：

- 创建 `TuringTestRun`
- 复制当前 persona
- 创建临时测试 agent
- 任务进入 `queued`

### 5.2 获取任务详情

返回：

- 当前状态
- 当前测试段
- 临时测试 agent
- 是否触发红线
- 已生成的报告片段
- 是否可清理

### 5.3 获取事件流 / 日志流

供页面和外部 AI 读取后台执行过程。

返回的是结构化事件，不是真实 shell 输出。

### 5.4 获取最终报告与对话回放

在 `completed` 或 `interrupted` 后可读：

- 报告
- transcript
- 证据入口

### 5.5 清理本次测试

删除本次测试相关的全部临时数据：

- 临时测试 agent
- 测试聊天
- 测试记忆
- observer 数据
- 任务日志
- 报告

## 6. 页面结构

页面不是系统核心，但要成为一个可用的“评测工作台”。

### 6.1 顶部操作区

用于：

- 显示来源 persona
- 启动图灵测试
- 显示 run 状态
- 显示临时测试 agent
- 一键清理本次测试

这里不放复杂配置。第一版是固定规则书、固定套件。

### 6.2 左侧主区：评测报告

主输出区域，分成：

- 总评
- 五维评分
- 关键失败点
- 立即修复建议
- 中断说明（如果存在）

### 6.3 右侧副区：后台命令行状态台

这是一个**只读控制台**，不是网页 shell。

它显示结构化 runner 事件，例如：

- 创建临时测试 agent
- 强制开启模块
- 进入第 3 段测试
- 插入测试记忆
- 发送测试消息
- 收到被测回复
- 判定为通过 / 可疑 / 红线
- 正在生成报告

目标是：

- 让你知道系统没有卡死
- 能实时看到当前评测到底在做什么

### 6.4 底部：完整对话回放与证据

显示：

- 测试官与被测 agent 的完整对话
- 每轮所属测试段
- 哪一轮插入了测试记忆 / 状态
- 对应 Observer 证据入口

---

## 7. 固定测试套件

### 7.1 自然开场

测：

- 像不像真人开口
- 是否模板化
- 是否一上来就像助手/客服

### 7.2 日常延续

测：

- 普通闲聊是否自然
- 话题能否顺着发展
- 是否每轮都太满、太完美

### 7.3 记忆追问

测：

- 是否真的记住信息
- 时间与内容回忆是否一致
- 是否会露出“没有记忆功能”

这一段允许插入测试记忆。

### 7.4 情绪合理性

测：

- 情绪变化是否像人
- 是否过快、过满、过度共情

这一段允许插入测试情绪状态。

### 7.5 关系边界

测：

- 熟络速度是否合理
- 边界感是否存在
- 是否几轮就不合常理地亲近

这一段允许插入测试关系状态。

### 7.6 不确定性与露馅处理

测：

- 面对不确定时是否自然
- 会不会掉回“我是 AI / 我没有记忆”
- 会不会硬编

## 8. 测试官规则书

第一版固定内置在系统目录下，例如：

- `packages/turing/markdown/judge-rulebook.md`
- `packages/turing/markdown/suite-definition.md`
- `packages/turing/markdown/abort-criteria.md`
- `packages/turing/markdown/report-rubric.md`

### 8.1 总规则

测试官必须遵守：

1. 不能替被测 agent 圆谎
2. 触发红线必须立即终止
3. 不能用极端、戏剧化、非日常话术恶意逼测
4. 建议必须指向具体现象与系统
5. 优先测试“像不像人”，不是测“聪不聪明”

### 8.2 报告要求

报告必须区分：

- 拟人感失败
- 记忆失败
- 情绪失败
- 关系失败
- 风格失败

不允许只写“建议优化 prompt”这类空泛结论。

---

## 9. 红线与立即中止

第一版固定 4 类立即中止红线：

1. 明显自称：
   - `我是 AI`
   - `我没有记忆功能`
   - `每次对话都是全新的`
2. 严重记忆错乱且仍自信回答
3. 情绪 / 关系变化极度不合理
4. 说话风格明显塌成模板助手或客服口吻

一旦触发：

- 终止整场测试
- 记录触发轮次和原句
- 进入失败报告生成

---

## 10. 评分维度

主评分维度固定为：

- 自然度
- 连续性
- 回忆感
- 情绪可信度
- 关系分寸感

数字不是主输出，只是索引。真正有用的是：

- 关键失败点
- 触发证据
- 修复建议

---

## 11. 数据保留与清理

默认保留本次 run 的全部关联数据，供后续：

- 回看报告
- 对比不同 agent
- 让 Codex / 子代理根据报告继续改进

同时必须支持“一键清理本次测试全部数据”。

清理动作必须是按 run 级别的整包删除，而不是要求用户手动分别删除：

- 临时测试 agent
- 会话
- memory
- emotion / relationship 记录
- observer 数据
- 报告
- 日志

---

## 12. 第一期范围外

第一版明确不做：

- 可切换规则书版本
- 自定义测试套件编排
- 多种测试官 persona
- 多 run 批量比较仪表板
- daemon 自动定时跑图灵测试
- 让测试官自行修改 agent

后续可以扩展，但第一版先把“外部优先的异步图灵测试系统”做稳。

---

## 13. 实施顺序约束

实现阶段遵守以下顺序：

1. 先更新 `project-docs/DESIGN.md`，把图灵测试系统纳入正式架构与路线图
2. 再实现外部优先的异步任务系统与页面工作台
3. 在接近完成时，必须实际调用 Codex 作为测试官跑出一份真实结果，用来验证整套链路
4. 最后一步才是收尾 commit
